1

📄 Os dois arquivos

A skill storm-research não é um programa complicado. Ela é uma pasta com dois arquivos de texto: o SKILL.md, que descreve o pipeline de 4 fases, e o report-template.html, que é o molde do briefing final. Mais nada. Toda a inteligência mora no texto desses dois arquivos.

O diagrama abaixo mostra esse aninhamento: o SKILL.md contém o pipeline inteiro (as 4 fases), e ao lado vive o molde que a Fase 3 clona para gerar o relatório.

A caixa azul grande é o SKILL.md — ele engloba todas as 4 fases (caixas ciano). A Fase 3 sai do arquivo de pipeline e clona o report-template.html para montar o relatório. Dois arquivos, um pipeline.

estrutura da pastao que você baixa
.claude/skills/storm-research/
├── SKILL.md              # o pipeline em 4 fases
└── report-template.html  # o molde do briefing final
📜 SKILL.md

As instruções em Markdown. Descreve as 4 fases, os prompts das 5 lentes e as regras de verificação. É o que o Claude lê e executa.

🎨 report-template.html

O HTML vazio com o design pronto. A Fase 3 preenche as seções; o CSS e a identidade visual não se tocam.

🟡 Novo aqui? — "skill" e "pasta da skill"

Uma skill é um conjunto de instruções que o Claude Code carrega quando você precisa dela. A "pasta da skill" é só um diretório com o nome da skill dentro de .claude/skills/. Tudo que está nessa pasta viaja junto — por isso bastam os dois arquivos.

2

🏷️ O frontmatter

No topo do SKILL.md mora o frontmatter: um pequeno bloco entre --- com três campos. Ele é o "cartão de identidade" da skill. O campo mais importante é a description — é por ela que o Claude sabe quando a skill deve ser acionada.

storm-research/SKILL.md (topo)frontmatter real
---
name: storm-research
description: Use quando alguém pedir para rodar Storm
  Research, aplicar o método STORM em um tópico… executa um
  pipeline em 4 fases: cinco lentes → mapa de contradições →
  relatório HTML → revisão adversarial + verificação.
argument-hint: "[tópico a pesquisar]"
---
name

O identificador da skill. Tem que bater com o nome da pasta.

description

O gatilho. Diz ao Claude quando usar a skill — em linguagem natural.

argument-hint

Dica do que passar como argumento: o tópico a pesquisar.

🎯 Por que a description é o coração

  • •O Claude lê todas as descrições disponíveis e escolhe a skill cuja description casa com o seu pedido.
  • •Por isso ela cita os gatilhos: "rodar Storm Research", "método STORM", "briefing verificado", etc.
  • •Você não digita um comando especial — pede em linguagem natural e a description faz a ponte.

🟡 Novo aqui? — "frontmatter"

"Frontmatter" é aquele bloco de metadados no topo de um arquivo Markdown, delimitado por --- em cima e embaixo. Não aparece no conteúdo; serve para o programa que lê o arquivo (aqui, o Claude Code) saber dados sobre ele.

3

🧳 Autocontida e portátil

A skill é autocontida: ela depende apenas das ferramentas nativas do Claude Code. Não há scripts externos, APIs, chaves ou serviços pagos para configurar. Você solta a pasta em .claude/skills/ e ela funciona — em qualquer máquina.

🤖 Agent

Dispara os subagentes general-purpose das lentes e dos verificadores.

🌐 Busca / fetch web

Usados dentro dos agentes para pesquisa real e verificação de fontes.

💾 Write

Salva o relatório final em storm-reports/.

✓ O que ela usa

  • ✓Ferramentas nativas: Agent, Write, busca/fetch web
  • ✓O report-template.html que vem na mesma pasta
  • ✓Só a pasta dentro de .claude/skills/

✗ O que ela NÃO precisa

  • ✗Scripts em Python/Node ou um build
  • ✗APIs externas, chaves ou serviços pagos
  • ✗Outras skills ou dependências instaladas

Por que isto importa

Portabilidade significa que você compartilha a skill mandando uma pasta. Quem recebe não instala nada — solta em .claude/skills/ e roda. É também o que permite levá-la para outros agentes (você vê isso na Trilha 3) e auditá-la, já que é tudo texto legível.

4

🎨 O template HTML

O segundo arquivo, report-template.html, é o molde visual. A regra de ouro está escrita na própria skill: clonar o template e preencher as seções — nunca recriar o CSS. O design já foi pensado; mexer nele só introduz inconsistência.

Limpo e profissional

Fundo claro, layout sóbrio, nível "tomador de decisão".

Montserrat / Roboto Mono

Tipografia fixa: título em Montserrat, código em Roboto Mono.

Destaque azul

Uma cor de acento só. Sem arco-íris de cores.

🧱 Clonar, não reinventar

Pensa no template como um formulário em branco: o trabalho da Fase 3 é preencher os campos (resumo, achados, referências), não redesenhar o papel. O CSS, as fontes e a paleta ficam intactos.

  • •Você muda o conteúdo das seções, não o estilo.
  • •Relatórios diferentes ficam com a mesma identidade — fácil de ler e comparar.

💡 Dica prática

Personalizar o template com segurança (sem quebrar a identidade) é assunto da Trilha 3, módulo "Customizar e estender". Por ora: trate o CSS como intocável.

5

💸 Custo: ~9–11 agentes por execução

Uma execução completa cria cerca de 9 a 11 agentes: 5 lentes na Fase 1 e ~4 a 6 verificadores na Fase 4. Isso é esperado e previsível — é o preço de cobrir vários ângulos e checar as citações. A conta abaixo mostra de onde vem o número.

5

Fase 1 — cinco lentes

Profissional Prático, Acadêmico, Cético, Economista e Historiador. Cinco agentes general-purpose numa única mensagem, em paralelo.

4–6

Fase 4 — verificadores

Cerca de 4 a 6 agentes, um por cluster de citações relacionadas. Cada um confere as afirmações contra a fonte primária.

=

Total: ~9 a 11 agentes

As Fases 0, 2 e 3 são feitas pela sessão principal, sem agentes. Por isso o custo não explode de surpresa.

⚖️ Não infle o pipeline

Mais agentes não é melhor. A skill diz explicitamente: não expandir além de 5 lentes nem usar mais de 1 verificador por cluster de citações. O ganho vem da diversidade de papéis, não da quantidade bruta.

6

🛡️ Proteções embutidas

A skill não é só um gerador bonito de HTML — ela carrega proteções escritas no próprio SKILL.md. Pesquisa real apenas; nada inventado; se um número não verifica, ele é rebaixado ou cortado; e a verificação da Fase 4 é obrigatória. Essas regras são o que separa um briefing STORM de um texto plausível qualquer.

✓ Promessa cumprida

  • ✓Toda lente e citação aponta para uma fonte real consultada
  • ✓Número que não verifica é rebaixado ou removido — nunca disfarçado
  • ✓A Fase 4 (verificação) é obrigatória; o banner de checagem precisa ser verdadeiro
  • ✓O relatório revela que o painel é autoral — convergência não é consenso do campo

✗ Red flags (não é STORM)

  • ✗Estudos, números ou URLs inventados para "completar" o texto
  • ✗Entregar sem rodar a Fase 4 de verificação
  • ✗Apresentar a concordância entre lentes como prova independente
  • ✗Inflar para mais de 5 lentes achando que melhora
